---
title: "Local-first data storage in GPT Image Playground"
description: "GPT Image Playground stores all task records and images in your browser's IndexedDB. No data ever leaves your device beyond direct OpenAI API calls."
---

GPT Image Playground is built around a local-first philosophy: every task record and every image you generate or upload is stored exclusively in your browser's IndexedDB database. Nothing is sent to any server other than the direct API calls you explicitly trigger. Close the tab, restart your device, or go offline — your history is still there the next time you open the app.

## What is stored

The app uses a single IndexedDB database named `gpt-image-playground` (version 1) with two object stores.

### The `tasks` store

Each completed or in-progress generation creates one `TaskRecord` entry, keyed by its `id`. A task record holds everything needed to display, reuse, or inspect a past generation:

| Field | Type | Description |
|---|---|---|
| `id` | `string` | Unique task identifier |
| `prompt` | `string` | The prompt text submitted for this task |
| `params` | `TaskParams` | The generation parameters you requested (size, quality, format, etc.) |
| `actualParams` | `Partial<TaskParams>` | Parameters the API reports it actually used, when different from requested |
| `inputImageIds` | `string[]` | IDs of reference images used as input, referencing entries in the `images` store |
| `outputImages` | `string[]` | IDs of generated output images, referencing entries in the `images` store |
| `status` | `'running' \| 'done' \| 'error'` | Current task state |
| `error` | `string \| null` | Error message if the task failed |
| `createdAt` | `number` | Unix timestamp (ms) when the task was submitted |
| `finishedAt` | `number \| null` | Unix timestamp (ms) when the task completed or failed |
| `elapsed` | `number \| null` | Total duration in milliseconds |
| `isFavorite` | `boolean` | Whether you have starred this task |

### The `images` store

Every image — whether uploaded by you or generated by the API — is stored as a `StoredImage` entry, keyed by its SHA-256 hash (see [Image deduplication](#image-deduplication) below):

| Field | Type | Description |
|---|---|---|
| `id` | `string` | SHA-256 hash of the image data URL, used as the primary key |
| `dataUrl` | `string` | Full base64-encoded data URL of the image |
| `createdAt` | `number` | Unix timestamp (ms) when the image was first stored |
| `source` | `'upload' \| 'generated'` | Whether the image came from a file you uploaded or was generated by the API |

## Settings storage

Your settings and generation parameters are not stored in IndexedDB. Instead, they are persisted in browser `localStorage` under the key `gpt-image-playground` using Zustand's persist middleware. This covers:

- **API settings** — `apiKey`, `baseUrl`, `model`, `timeout`, `apiMode` (`images` or `responses`), and `codexCli`
- **Generation parameters** — the current values of size, quality, output format, compression, moderation, and batch count (`n`)

These values are restored automatically when the app loads, so your preferred configuration is always ready on the next visit.

## Image deduplication

When you upload or generate an image, the app computes its SHA-256 hash using the Web Crypto API (`crypto.subtle.digest`). On browsers where the Crypto API is unavailable, a deterministic FNV-style fallback hash is used instead (prefixed with `fallback-`).

The hash becomes the image's `id`. Before writing to IndexedDB, the app checks whether that `id` already exists. If it does, the write is skipped entirely. This means:

- Uploading the same file twice results in only one stored copy.
- Generating an identical image twice also stores it only once.
- Both input and output images share the same deduplication pool.

## Memory cache

When you add images to the input panel before submitting a task, the app holds them in an in-memory `Map` keyed by image `id`. They are **not** written to IndexedDB at this point. The write to IndexedDB happens only when you click **Generate**, just before the API request is sent.

Similarly, when the app loads all tasks on startup, it pre-populates the memory cache from IndexedDB. Subsequent reads for those images (for example, when rendering task cards or opening the detail view) are served from the cache, avoiding repeated IndexedDB reads.

When a task finishes, the input images used for that task are removed from the memory cache, since they are now safely persisted to IndexedDB and can be loaded on demand.

## Orphan cleanup

Each time the app starts, `initStore` runs a cleanup pass:

1. All task records are loaded from IndexedDB and their `inputImageIds` and `outputImages` arrays are collected into a set of referenced IDs.
2. All images in IndexedDB are scanned. Any image whose `id` is not in the referenced set — and is not currently in the input panel — is deleted.

This prevents storage from growing unboundedly over time as you delete tasks or clear history. Deleted tasks take their associated images with them automatically; orphan cleanup is a safety net for any images that slip through.

## Clearing data

The **Clear all data** button in **Settings** wipes everything in one step:

- All records in the `tasks` IndexedDB store
- All records in the `images` IndexedDB store
- The in-memory image cache
- The current input panel images
- All settings and generation parameters (reset to defaults)

<Note>
  Clearing browser storage directly — for example via **DevTools → Application → Clear site data** — will also erase all tasks, images, and settings stored by the app. This action is irreversible. Export your data before clearing browser storage to avoid losing your history.
</Note>
